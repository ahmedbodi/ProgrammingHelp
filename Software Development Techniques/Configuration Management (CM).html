Configuration Management (CM)

Created:05/05/2016 11:45
Updated:11/05/2016 11:31
Author:ahmedbodi@crypto-expert.com

Configuration Method is a collection of procedures used to manage aspects of the quality in an evolving system.

It is principally concerned with the control of change and visibility of software products

The CM procedures are usually described by a set of standards contained in an organisations Quality Assurance Manual, and are themselves subject to change and improvement.

Why do we need a Configuration Method:
     - Many software projects are characterised are:
          - Large size (very many artefacts)
          - Long Term (sometimes maintained over decades)
          - Subject to continuous change and development
     - Software modules may exist in several versions and releases
          - Each version and release will have its own supporting documentation, specification, design, testing.

     - These artefacts are heavily inter-related
          - Changes to one may affect literally hundreds of others
     - In such an environment, tight control of these artefacts and their relationships in neccessary to avoid chaos

Configuration Management in Practice
     - From a project management perspective, CM allows a snapshot to be generated of the state of every aspect of the development at any given time.
          - Useful for measuring progress
          - If things go wrong, it allows the whole project to be re-wound to last known good state
     -  From a developers point of view, it facilitates communication of changes taking place elsewhere in the project that directly affect them and their work
          - E.g. is a designer using the latest version of the spec
          - E.g. is a programmer using the latest deign document

The principles of Configuration Management:
     - Identification of Configurable Items (CI)
          - The unique (and uniform) identification of every document and program module that is brought under configuration)
               - In all their versions, editions, revisions and releases
     - Change Control
          - Mechanisms to ensure changes are properly engineered, tested and reviewed before being accepted
     - Status Accounting
          - Recording and reporting on all aspects of CI's
              - History of items, relationships between them etc
     - Configuration Assurance
          - A quality procedure to monitor the CM effort and ensure it is working effectively

Configuration Management Planning
     - The starting point for a CM plan should be a set of standards adapted to fit the particular project
     - The plan should
          - Define what it is to be managed, with an identification scheme
          - Define who is responsible for CM procedures and submitting controlled entities to the CM team
          - Define policies for version management and change control which all team members must use
          - Specify the tools to be used, and procedures for using them
          - Describe the structure of the CM database and the information that should be contained with it

Configuration Item Identification
     - The first step in producing an identification scheme is to determine which products or artefacts should be controlled
          - Typical products include
               - Quality Control - Including CM procedures
               - Project plans
               - Requirements
               - Test Documentation
               - Test Plan/Scripts/Data/Results
               - Analysis Documentaiton
               - Design Documentation
               - Program Code
               - User documentation
               - Training Material

Configuration Item (CI):
     - The scheme should guarantee the ID for each CI is unique
          - This follows from a structured, hierarchical breakdown of the product
     - The scheme should be flexible, so that new classes of artefact can be added later on as the project develops
     - The ID should be able to be embedded in the artefact
          - The ID should be able to embedded in the artefact 
               - E.G. a comment in the code
               - If existing artefacts are inherited from other projects, ideally their new ID should correspond to their existing one so they don’t have to be re-written
     - IDs should be meaningful to the project team

Change Control
     - Change control is about managing the evolution of individual components in the system to maintain overall integrity
          - This is achieved through the mechanism of Baselines
     - A baseline is a product that has been formally reviewed and agreed upon
          - The baselines serves as the basic for further development
               - Throughout the life of a project, baselines will be made at regular intervals
               - After “baselining”, any subsequent changes to a product are subjected to the same quality procedures again.
     - Definitive versions of all products are available at all times

Version Control tools allow documents / products to be checked in and out of a central database
     - The CM team can track and control changes

The check-in process can include integrating changes
     - Proposed modifications are analysed and merged with the existing product

Using CM for defect management
     - After initial development, software change is driven by errors
     - When a test fails (or when a user reports a bug), a defect is recorded.
     - The defect (fault) is investigated, and where appropriate, software change requested is raised.
          - Change requests will be considered in terms of its impact, risk, cost and urgency before being assigned to a developer to be implemented
     - Once a change request is approved, the developer will check out the code, implement changes, run regression tests according to the test plan, check in the changed code and submit the changes for approval
     - The changes can then be incorporated into the next baselines
     - Details of the fault, the analysis and any changes made are all submitted to the CM database
     - The status of fault reports and change requests in the database are monitored through status accounting by the project management.
          - Also metrics can be generated based around
               - The types of fault discovered
               - The severity
               - How they were discovered
          - Which can help to improve the development process for the future
               - E.G. need more testing, analysis, personnel
