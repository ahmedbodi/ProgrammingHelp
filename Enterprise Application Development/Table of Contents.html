Enterprise Architecture Framework

Map
Businesses use IT to achieve goals which is to make money. For this processing information is a key activity and shared information is accessed by many applications. 

An architecture defines process, information, IT & application standards, ensures the applications and processes can work together, ensures the consistency and security of the architecture and reduces costs of maintenance and purchasing. An architecture framework helps plan an architecture. 

A business or business process architecture defines business strategy, rules, organisation and processes. 

An applications architure is a plan for the individual applications, their interactions and relationships to the business processes. 

A data architecture is the structure of logical and physical data and management resources

A technology architecture is the logical software and hardware capabilities including support deployment of business, data and application services and also includes IT infrastructure, middleware, networks and standards. 

Creating an enterprise architecture is complex, it has many technical, business and human problems. An architect framework can guide the developer with a static snapshot of features and viewpoints to ensure that nothing has been missed out. It can also be dynamic with development processes and lifecycle to speed up development. It is a tool for designing an information system with a set of building blocks, rules for combining the building blocks, a common vocabulary, recommended standards and compliant products for implementing building blocks. 


Cloud Computing

Map
Cloud computing means accessing computing remotely across an internet connection not on a local PC or a portable device. A model for enabling convenient on-demand network access to a shared pool of configurable computing resources such as networks, servers, storage, applications and services that can be rapidly provisioned and released with minimal management effort or service provider interaction. It abstracts away infrastructure complexities of servers, applications, data and heterogenous platforms. 

Cloud Service Models:
    - Software as a service where a person can use a providers applications over a network. This is usually a thin client interface where only user specific configuration settings are changed.
    - Platform as a service where customer created applications are deployed to a cloud. Its possible to control deployed applications and environment configuration
    - Infrastructure as a Service
        - Rent processing, storage, network capacity and other fundamental computing resources. Its possible to control OS, storage, deployed applications and possibly network components such as firewalls and load balancers

Software as a service
Examples of software as a service are customer relations management platforms, financial planning, human resources and word processing. Examples of commercial applications which belong in this category are google docs and windows live. 

Platform as a service
This provides a programming platform and tools such as java, python and .net and building blocks and API’s for building cloud based applications and services. Examples of this are amazon elastic beanstalk, microsoft azure websites and google app engine.

Instrastucture as a service
This is where infrastructure is provided such as hardware, OS and communications. There is access to part of the stack such as OS access, firewalls, routers and load balances. Commercial examples of this are amazon elastic cloud computing or digital ocean. 

Traditional data centres have costs such as the purchase and maintenance of hardware, utilisation and energy costs. 

The benefits of cloud computing are:
    - Cost savings - lower cost of ownership and pay for what you use
    - Security
    - Reliability and availability since you only pay for what you use
    - Reduce IT management responsibility due to automated update policies
    - Performance
    - Flexibility - increased agility in software deployment due to faster app rollout times and increased capacity
    - Scalability by allowing for unexpected resource loads and on demand renting

Risks of cloud computing
    - Security
    - Downtime due to the possibility of massive outages
    - Dependency, potentially tied to provider, pricing changes

Advantages of Cloud Security
    - Cloud uniformity simplifies testing, auditing and automated security management
    - Redundancy and disaster recovery due to rapid reconstitution of services, enables availability and advanced honey pot capability (fake server as bait)

Cloud Security Issues
    - Privacy - multi tenancy sharing and international data transfer
    - Risk of attack - attraction to hackers due to being a high value target, trusting vendors security model, loss of physical control and indirect administrator accountability
    - Obtaining support for investigations

Security in J2EE

Map
A group is a collection of users
A realm/security domain is an area controlled by a security policy or a collection of users and groups controlled by the same authentication and authorisation policy. 
A role is a set of permissions to access a specific set of resources. 
A principal is an entity that can be authenticated and potentially has rights e.g. check ID
A security attribute is a value associated with a principal e.g. expiry date, password, etc
Credentials is information used to authenticate a principal e.g. password.

Security can be applied at the application level which can be managed by the program or the container to the set of programs/beans/servlets which is necessary to control access to actions. It can be applied at the transport level to provide privacy but not access control to hide data during transfer and at the message level to encrypt data before sending. 

Application level security is provided by components or containers and is specific to the needs of the application. Firewalls can enhance application protection by protecting communications and resources from attacks. Application level security is easy to implement and configure and provides fine grained control of functions and data. It only protects in the application environment and needs additional transport layer security to hide passwords and data during transmission. The advantages of application level security is that the security is tailed to the application and is more precise. however security attributes are managed only in the application and other applications may not maintain security and support for multiple protocols increases risk since they can’t be used on its own. In addition to this a failure in the application may allow exposing all the data. 

Transport layer security uses SSL to secure the communication.  The advantages of this is that it is simple, well understood and a standard technology. It applies to the message body and the attachments however it is highly coupled with the transport layer protocol and has an all or nothing approach to security, portions of the message cannot have TLS applied and it is not an end-to-end solution only point to point with the protection removed at the destination. 

Secure Socket Layer is the process of sending an initial handshake using public key encryption where the server sends a certificate containing its public key and then the client sends the crypt data using the servers public key. Only the server can decrypt it with the servers private key and this allows the server to agree on a symmetric session key. To exchange data, errors are checked using stream based communication with the symmetric key to encode. 

Message level security is where the information is stored in the message securely. The information travels with the message or attachment with encrypted parts opaque to intermediate nodes using end-to-end security. The advantages of this is that the security stays with the message at its destination and the security can be applied to portions of the message. It can also be used with intermediaries over multiple hops and is independent of the application or transport protocol however using message level security is complex and adds overhead to processing. 

Specifying security:
Declarative security is where security is defined using metadata such as annotations and deployment descriptions to override the annotations.
Programmatic security is where the application makes security decisions and allows finer control than declarative security alone. 

To secure enterprise beans, first the security view is specified with a set of security role which is a group of permissions that certain users require to access parts of the application such as logical rules in the organisation like user, manager, administrator. Then the methods that the role can access are defined such as specific endpoints or pages and finally an authorisation mechanism is specified which illustrated the way the identity of a user will be confirmed. 

The possible authentication mechanisms include:

HTTP Basic Authentication 
 The server requests username and password, which the server looks up in a database of authorised users

Form Based Authentication
The server asks the user to input their username and password in a form on a page and then looks up the details inside the database

HTTPS Client Authentication
The server checks the client’s public key certificate which is a more secure method of authentication. This is done using SSL which provides encryption, server authentication, message integrity and optional client authentication for a TCP/IP connection

Mutual Authentication
The server and the client authenticate each other. There are 2 types of mutual authentication. Certificate based mutual authentication and username/password based authentication. 

Public key certificates are the digital equivalent of a passport. it identifies an entity by providing their public encryption key. it is provided by a trusted organisation such a certificate authority and uses public key encryption to verify the certificate authority and to define the identity.

With security issues, flaws may have no effect in normal use, the attacker must create the failure by creating specific conditions. the flaws can be in the protocols, operating systems, applications, web server, network traffic, and ease of downloading software at work etc. 
When developing secure systems, the design should be based on security principles with systematic design based on a good practice with people taken into account. A review of the design and architecture should also be considered to discover and correct potential security problems. 

Designers must consider:
    - Identification
    - Authentication
    - Authorisation and access control
    - Administration
    - Attacks
    - Detection
    - Audit (keeping a log of things that have occurred)

Design principles for security:
    - Open design (assume attackers have the sources and the specs)
    - Fail-safe defaults (deny access by default)
    - Least privilege (no more rights than need)
    - Simplicity
    - Check everything every time
    - Least common mechanism, (avoid sharing resources between users)
    - Manage security policies centrally
    - Design security into the architecture (maximise usability and minimise modifications)
    - Maintain a single, accurate date/time
    - Perform a risk assessment
    - Handle varying security needs
    - Implement a security awareness program

Threat modelling approaches
    - Asset driven threat modelling - list assets and how they might be attacked
    - Attacker driven - possible attackers and their approaches
    - Design driven - identify components and trust boundaries - e.g. sockets, remote procedure call


STRIDE Threat Model
Spoofing  which affects authentication
Tampering which affects integrity
Repudiation which affects Non-Repudiation
Information disclosure which affects Confidentiality
Denial of service which affects availability
Elevation of privilege which affects authorisation

The STRIDE modelling approach involves breaking the system down into relevant components, then modelling the system with Data Flow Diagrams to analyse components for susceptibility to STRIDE and then ways to mitigate the threats. The threats which emerge when the systems are combined into larger systems are also considered which could occur when assumptions about a subsystem is not considered such as a system which isn’t designed for internet use is exposed to it. 

To ensure that security is strong, first is to ensure that strong policies are in place at an organisation and that they are followed. Then to ensure that during the recruitment and training that the personnel are following the processes such as strong passwords and not sharing them. Finally it includes ensuring that security is designed in the application and that bugs in the application are eliminated such as buffer overflow, SQL injection and HTML injection. 
This can also include ensuring file protection and authentication is used in the operating system, network and hardware ensuring theft is not easily done and that in the IT management department administrator passwords are secure and strong and that backups are regularly done and stored in an easy to access place but securely kept. 

Pattern

Map
A pattern is a reusable solution to a recurring problem. The components of a pattern are:
    - Name
    - Problem description
    - Forces - relevant factors
    - High level solution
    - Advantages and disadvantages
A pattern should explain why and how. It captures a chunk of knowledge using the design expertise and helps people understand design problems. It also helps to avoid re-inventing solutions to common problems and avoids the developer making subtle errors. 

The patterns that can be used at different levels include”
    Business Patterns
        - Common user, business and data interactions
    Application Patterns
        - Partitioning of application logic and data
    Code Design Patterns
        - Interaction between program components

To design with business patterns involves reviewing the current environment and goals including the organisation goals and influences, the business requirements, the organisation’s IT goals and environment and finally the functional and non-functional requirements. After this it involves, outlining the business solution, picking appropriate business patters, picking appropriate application/code patterns and designing and implementing the details. 

Business patterns:
    - Self service (user to business)
        - Useful when users must directly interact with business processes, these business processes must integrate with existing business systems and information, these business processes must be accessible in a common, consistent way through various channels. This pattern allows users to access and change their information by interacting directly with core business systems. 
        - The components involved in this are the users with a web browser allowing them to access the solution from anywhere on the network. A network based on TCP or other technologies. An enterprise system run by the business and the interactions which provide business functions to the users.
    - Collaboration (user to user interaction)
        - Useful when one or two way interactions among users such as asynchronous collaboration (email, video sharing), Interactive synchronous collaboration (chat rooms, instant messaging) and workflow interactions (designers -> programmers -> testers -> clients)
        - The components involved in this are multiple users, a network, a set of collaboration services including a directory so users can find each other and different types of data. It also includes a set of interactions such as one to one, one to many and many to many workflows involving users or applications. 
    - Extended Enterprise (business to business)
        - Useful when organisations must work together with the details defined by trading partner agreements and service level agreements. 
        - The components used in this are the business entities which interact across the network, a network such as a WAN or internet, business rules governing the integration between business entities, workflow rules such as steps, data flow and formats and interactions that perform an agreed process with loose coupling that hide private processes and implementation allowing the implementation to change more easily. 

Code Design Patterns:
    - Separate UI From Main Processing (Application-Controller / Model-View-Controller)
        - Application controller involves the creation of a controller object to manage page flow and navigation logic, which manages state across user interactions and may interact with the model to perform business actions or to define the flow based on he model state.  With this pattern there are more components to manage and a way to create and connect views and controllers are needed which could be dependency injection. 
        - MVC allows separating the data model, presentation and actions based on the user input, the model manages behaviour and data of the application domain and provides data about its state to the view and can change state on request. The View manages the display of information. The controller interprets input from the user and tells the model or the view to change as appropriate. The structure of MVC means that the view and the controller depend on the model whereas the model does not depend on any of them. The model can be built and tested independently of the visual presentation. The separation between view and controller is secondary and many user interface frameworks implement the roles as one object. In web applications the separation between view and controller is well defined. Variants of this include the passive model where one controller manipulates the model exclusively, another is the active model where the model changes state without the controller. The advantages of MVC are that it makes testing easier due to the separate data store, displaying and updating, the ability to have multiple views and to change components. However there is a greater complexity and the user interface code is more event driven and the cost of frequent updates may increases resulting in slow rendering of the views.
    - Decouple worker and watcher (observer)
        - This pattern is useful when the source object must tell viewers of changes and the source should not know details of others. Additionally there is direct coupling which reduces the reusability of classes. 
    - Separate action from coordinator (Inversion of Control)
        - This patten is used instead of a program calling services the components are called by a container resulting in a greater flexibility. This is related to the dependency injection pattern. This is useful when a client class depends on components whose type is specified at design time. This makes it difficult to test each class in isolation when direct references are used to dependencies. 
    - Separate component finder from user (Dependency Injection)
        - This pattern involves dependencies not being called directly, a builder object is used to obtain valid instances of dependencies and passing them to the client. Constructor is used to pass parameters to contractors to inject dependencies and setter injection is used to inject dependencies where the client has exposed setter properties 
        - The problem with this is that there are more components to manage and the complexity of the injection framework can increase resulting in source code that is harder to debug.

HTTP Verbs:

Post: submit data from a form, pass parameters, may change server state

Defining Restful Services

Map

WSDL2 is an implementation of SOAP but using REST which can describe various transport protocols and is an agreed standard however more complex. 

With a resource orientated architecture there is a base design around resources and limited actions. Every resource is a directly accessible distributed component identified by a URI and linkable to other representations which is handled through a standard common interface (API). . information is passed about the current state of the resource and a service offers CRUD access to a resource. 

REST and ROA is the alternative to SOA and web services. 

To create a RESTful web service
the conceptual entities to be exposed as services are identified e.g. /parts/1 or /exam/sciences/computer/1

classify resources and interactions, use GET to request representation and POST/PUT/DELETE to modify
put links in representations to  link to more information
describe how services are invoked using WSDL2 or WADL

REST vs SOAP philosophies
REST:
    - applications publish their data on the web through the URI
    - resource orientated

SOAP:
    - applications use the web to interact but remain outside it
    - service orientated. 

REST vs SOAP technology
REST:
    - few operations (generic interface) on many resources
    - consistent naming for resources URL
    - focus on scalability and performance of large scale distributed hypermedia systems
    - composition with applications from different sources

SOAP:
    - many operations (service interface) on few resources
    - no standard naming mechanism
    - focus on design of integrated (distributed) applications
    - composition via business processes

REST vs SOAP protocols
REST:
    XML in - XML out with POST
    URI in - XML out with GET
    Self Describing XML    
    HTTP only
    Synchronous
    Do it Yourself for reliable message delivery and distributed transactions
    Security based on HTTPS / SSL

SOAP:
    SOAP in - SOAP out (with POST)
    Strong typing (XML schema)
    Transport indenpendent (multiple protocols)
    Synchronous and Asynchronous
    Various protocols built on top of SOAP
    Has own security

REST vs SOAP design approaches
REST:
    identify resources to be exposed and nice URL’s to address them
    identify read only side effect free and modifiable resources
    relationships between resources correspond to hyperlinks that can be followed
    implement and deploy on web server

SOAP:
    list operations in the WSDL doc
    define a data model for the content of the messages (XML Schema data types)
    choose an appropriate transport protocol, define the Quality of Service, security and transactional policies
    implement and deploy on the web service container


URI's

Map
A URI represents resources. They generally look like nouns. They should be intuitive, easy to guess / self documenting. A straightforward with an easily understood structure and human readable. 

URI’s are used to define a directory structure-like URI’s with a hierarchical path and branching from it are subpaths which expose the services main areas. If the URI is logical, then the implementation technique must not be revealed so the implementation can be changed without affecting clients. URI’s must stay static and query strings should be avoided. 404 pages should not be used instead a default page should be shown. Error responses should be human friendly with examples and hyperlinks and use standards such as an XML schema to document and check acceptable formats. 

Sessions

Map
Stateless sessions is where each request from a client to a server must contain all the information necessary to understand the request and cannot take advantage of any stored context on the server which is necessary to have caching. To achieve stateless caching, the server generates responses with links to other resources and indicates if they are cacheable. This will improve performance by reducing duplicate requests and the server will do this by adding a Cache-Control/Last-Modified date in the HTTP response header. After this the client application will use cache-control to decide when to cache the source and send a last-modified state in a conditional to see if the resource has changed which saves bandwidth

SOAP/WSDL Approach

Map
The SOAP/WSDL approach uses Remote Procedure Call for the protocol for communicating data over HTTP. 

WSDL describes procedures and parameters and SOAP wraps the call and parameters and result. HTTP then transfers the SOAP envelope with call and parameters and then returns a SOAP envelope with result. 

Web Services

Map
A web service is a software system identified by a Unique Resource Identifier (URI) basically a URL. They have public interfaces and binding which are defined in XML. The definitions can be discovered by other systems and other systems can interact with the service using XML based messages or internet protocols such as HTTP.

With a Service Orientated Architecture (SOA) aims to structure a complex system around service, building it from independent services. 

Web services are defined by a contract. Clients don’t need to know how a service is implemented. They aren’t tied to proprietary architecture, protocols or software. Rather they use standardises methods to access services. Communications are done in HTTP and data is represented in XML. XML is also used to describe services and parameters. 

Publishing a service
To publish a service a Universal Description, Discovery and Integration. This is published to a directory service (registry). Businesses register and search for web services. and these web service interfaces are described by WSDL. 
UDDI is a web service which is communicated with via SOAP. It also supports distributed registries. 

Describing a service
To describe a service the Web Service Description Language (WSDL) is used which uses XML. This describes the interface of a web service, the possible interactions with the service such as the methods, arguments and the results. It also includes the data formats such as the structures, fields and types. Finally it includes the address of the service and the communication protocols that can be used to exchange the data. 
The benefits of WSDL is that an interface contract is a description, the services are defined not the implementation like a java interface. The ability to write format descriptions allows the usage of tools such as syntax checkers and code generators. Enterprise Java Beans
An EJB is a reusable portable component which encapsulates business logic e.g. a method to update customer data in a database. These are developed as POJO’s, which communicate with the container by metadata. The developer focuses on the business logic and the container handles common complicated issues such as transaction support, security and remote object access.  The important concepts in EJB’s are session beans which manage business logic of interaction with the client which can be stageful or stateless, message beans that wait for jobs and perform them and entity classes which manage system data

Session beans are associated with a client and perform actions on the server e.g. handle student application. A stageful session bean maintains information across calls during the bean lifetime. Session beans are short-lived, participate in transactions and don’t survive server crashes. 

Entity classes manage persistent data and are identified by a primary key. These are identified by the @Entity annotation. These represent persistent data from the database which are sharable across clients and represent business logic for a persistent entity.  Each entity class is persistent, allows shared access by multiple clients, have primary keys and participate in relationships. Entity classes can insert, update, select and remove data. With Bean Managed Persistence (BMP), the programmer writes code to access the data source which is more flexible but more work. With Container Managed Persistence (CMP), the container handles database access for the entity bean. The ben isn’t tied to a specific data source. The entity bean can be deployed in different containers with different data sources. 

Java Persistence API is an api which uses entities which are POJO’s with no special methods. It uses standardised object relational mapping with the specification defining default values for them. They support inheritance and polymorphism just like objects with support for Java Persistence Query Language and SQL. The JPA supports pessimistic locking and optimistic locking and has an EntityManager API for performing CRUD Create, Read, Update, Delete queries on entities. 

Message driven beans allow applications to use asynchronous messages. This means communication is done in a way which does not block the rest of the app, resulting in increased usability and speed. The container handles most of the administration and the two models for delivering messages are queue and topic. These are annotated with the mark @MessageDriven. They implement the java.jms.essageListener interface. The queue method is a point to point delivery service with many senders and receivers however the message is consumed by one bean. With the topic delivery service, there can also be many senders and receives but th message is consumed by all subscribing beans due to it being a publish-subscribe delivery format. 

The advantages of EJB are that it can be deployed in any vendors container since the bean can rely on the container to manage lifecycle, persistence, transaction, distribution and security. The transaction semantics are defined declaratively in annotations rather than programatically. Additionally EJB is highly scalable since beans can be distributed across multiple servers. client locations do not need to know the location of the EJB’s and the data sources can be changed easily resulting in a reusable bean where the business logic can be written one and reused.

Service Orientated Architecture

Map
A service orientated architecture means to build a system from components with defined interfaces. Each interface is a contract

Each component provides a service. Each component doesn’t care about how its implemented but uses a standard approach to access the service. Applications use services to provide loose coupling between provider and consumer.Create applications by combining services differently. SOA is the process of creating distributed services provided by different owners. 

SOA is defined as a system organisation using protocols and components to allow application facilities to be provided and consumed as sets of facilities published at a granularity relevant to the consumer. Services can be invoked, published and discovered using standards based interface. 

SOA is used to avoid duplication during the repeated development and data duplication It enables the consumers and the providers to interact across the different technology and different systems. The services should increase business agility by being flexible, supporting reconfiguration and reuse. This is done by using loose coupling since the dependencies are minimised an explicit stated. Stateless services not conversational interactions and hiding information. Also by ensuring that services are cohesive, complete and consistent. 

Its important that the right level of abstraction is used, keeping as course grained as possible but ensuring that the right level of relevance, consistency and completeness is kept. This ensures that atomic services can be composed into higher level, fully fledged services. 

The reference model for SOA is the OASIS reference model and architecture. 

The advantages of SOA include:
Reduced costs by re-using code and hardware/system independence. Flexibility through rapid application development by using tested services. Reliability due to services that are thoroughly tested. It can also hide legacy applications behind the services

The disadvantages of SOA include:
The added overhead can affect performance and it is also difficult to provide and find the right services. 

OASIS Reference model

Map

Service:
A way to access one or more capabilities. Uses a prescribed interface with data and behaviour models require to use the service. The information required by consumers to decide if it meet their needs. Bound by constraints and policies. The provider may not know the consumers or how they use it. the implementation is hidden but the interface should not be. Invoking a service has one or more real world effects. 

Service Descriptions:
 Functions and technical requirements, constraints and policies and access methods. Description syntax and semantics must be widely accessible and understandable.

Visibility
Ability of consumers to find providers and interact. This is implemented by common protocols, standards and technologies. 

Execution context
Specific circumstances surrounding an interaction. this means the protocols, encryption and other information regarding the interaction. This may affect how the service is involved.

Real world effect
The purpose of using a service which may change data in an enterprise and beyond by communication between services.

Interaction 
this means using a service which is an act which affects something else for example an exchange of messages calling a service and then returning the results. 

Policy and constraints
A policy is a constraint on the use of data or a service. tis can be the security and the quality of service. A contract is an agreement between participants. a policy subject to negotiation, service contracts ay be of any duration.

Providing and consuming services to realise mutually desirable real world effects.

Transactions

Map
A transaction is a group of operations performed as a single chunk to ensure consistency despite any errors. The transaction fails unless all operations succeed. If any operation fails all are rolled back. Once the transaction has completed the results are permanent such as transferring money between accounts. These transactions have properties called ACID.

Atomicity
    - All or none of the actions are performed

Consistency
    - Afterwards the database is in a permitted state

Isolation
    - Overlapping transactions can’t see each others changes, only before and after state 

Durability 
    - Results permanent even if the system then fails. 

To perform a transaction, the transaction is started, any operations are carried out and then the transaction is committed or rolled back either due to a failed operation or because the application decides not to. 

Distributed transactions means information is held on several machines which means consistency needs to be maintained which is hard. Have to cope with failure of a computer, communication problems and ensuring everyone commits or aborts so a controller is needed. 

The participants in a transaction its the application, the resource managers which store information such as the databases and the transaction processing monitor which co-ordinates the resource manages to enforce ACID. 

Container managed transactions automatically create the transactions using the container, the beans do not need to think about transactions except at the design time when the designer identifies whats in the transaction and to force roll back 

In bean managed transactions, the bean creates and commits or rolls back transactions. This allows finer control, transactions are only used when needed due to the added overhead but more errors are more likely

In a suspended transaction, the second transaction is independent of the first, if it commits, it stays committed even if the first aborts. 

In a nested tranaction, if the second transaction aborts, it can be resumed without aborting the first one, If the first transaction aborts, the second one is removed even if committed 

In such an implementation, mutual exclusion is needed for isolate, it uses locks to prevent simultaneous access to a resource to prevent 2 processes writing to a record. A read lock allows readers to access the data however a write lock, ensures single access. When this is used there is a risk of  a deadlock 

Optimistic transactions is where any action is taken first, then check if the data has changed before committing, the data can be rolled back then re-done, this means less time spent locking but it can be expensive to roll back

Pessimistic transactions is where the database record is locked to ensure no one else can read and this can be expensive as locking can take time. 

When doing transactions it is essential to avoid user interaction, keep transactions short to avoid risk of overall and test controllers under heavy load.
